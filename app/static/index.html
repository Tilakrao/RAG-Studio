About

Built for engineers who want to understand RAG, not just use it

RAG Studio is an open-source, 100% local evaluation workbench for Retrieval-Augmented Generation pipelines. It was created because most RAG tools hide the internals — you feed them a document, they give you answers, and you have no idea whether cosine similarity or BM25 would have worked better, or whether a cross-encoder reranker actually helped your corpus.

Every component is swappable and every comparison is transparent. Upload a PDF, configure multiple pipelines with different loaders, chunkers, and embedding models, then query them side-by-side. The metrics, scores, and chunk text are always visible so you can build intuition about what really moves the needle.

Because everything runs locally, there are no API costs, no data privacy concerns, and no external dependencies after the initial model download. Your documents never leave your machine.

Search Methods

7 retrieval strategies, side-by-side

Dense, sparse, and hybrid — compare them on your actual corpus to find what works

Dense
Cosine Similarity

Default. Measures angle between query and document vectors. Best for semantically rich text where absolute magnitude is irrelevant.

Dense
L2 Distance

Euclidean distance in embedding space. Sensitive to vector magnitude — useful when embedding norms carry signal.

Dense
Dot Product

Raw inner product. Fastest similarity; works well when vectors are unit-normalized by the model.

Sparse
BM25 Keyword

Classic TF-IDF variant. Excels on exact keyword matches, technical terms, codes, and named entities that embeddings may miss.

Hybrid RRF
Cosine + BM25

Reciprocal Rank Fusion of cosine and BM25 result lists. Captures both semantic intent and keyword precision.

Hybrid RRF
L2 + BM25

RRF fusion using L2 distance for the dense leg. Often outperforms cosine on models without explicit normalization.

Hybrid RRF
Dot Product + BM25

RRF fusion using dot product dense search. Try this when the embedding model uses dot-product as its native metric.

How Reciprocal Rank Fusion works

RRF fuses two ranked result lists without needing score normalisation. Each document gets a score of 1 / (k + rank) from each list (k=60 by default), and the scores are summed. Documents that rank highly in both lists float to the top; documents absent from one list still get a partial score if they rank well in the other. This makes hybrid search robust to the scale differences between cosine scores and BM25 term weights.

How to Use

Get started in 5 minutes

From starting the server to comparing RAG configurations — the complete step-by-step workflow

01

Start the server

Activate your virtual environment and start the FastAPI server. Always use --workers 1 — ChromaDB PersistentClient is single-process only.

.venv\Scripts\python.exe -m uvicorn app.main:app --host 127.0.0.1 --port 8000 --workers 1
02

Open in browser

Navigate to http://localhost:8000. You'll see this landing page. Click Launch App (top-right or footer) to open the pipeline studio. The app is localhost-only by design — the Host Guard middleware blocks all other origins.

03

Upload a PDF document

In the Documents tab, drag-and-drop any PDF or click "browse files". Files are validated via magic-bytes before storage. Scanned PDFs (no selectable text) are supported — the OCR loaders handle them automatically via PyMuPDF + Tesseract.

04

Configure pipeline components

Go to Pipeline Studio. Select your document, then choose a Loader, a Splitter (with chunk size / overlap), and an Embedding Model. Each component shows its configurable parameters dynamically. Add up to 4 different configurations to the Comparison Basket.

Tip: Start with pypdf + recursive_character (500/50) + all-MiniLM-L6-v2 as a baseline, then vary one component at a time to isolate impact.
05

Run pipelines & watch the step log

Click Run All to process all basket configurations in parallel. Each pipeline card shows a live step log: Loading → Splitting → Embedding → Storing — with per-step timing and chunk counts. Completed pipelines cache their ChromaDB collection for instant reuse.

Tip: The Run History tab shows all past runs. You can expand any row to inspect individual chunks.
06

Search, rerank & compare

In Search & Evaluate, select one or more pipelines, type a natural language question, pick a search method (dense / sparse / hybrid RRF), optionally apply a cross-encoder reranker, and click Search. Results appear side-by-side with similarity scores and chunk text. Use Pin to freeze one result set and compare it against a second query/config.

Under the Hood

How this application works

The technical mechanics that make RAG Studio fast, reliable, and reproducible

Idempotent Pipeline Hashing

Every pipeline (loader + splitter + embedder + doc_id) is deterministically hashed with SHA-256 into a unique ChromaDB collection name. Re-running the exact same configuration skips all reprocessing and returns the cached collection instantly. Benchmarking is cheap — you only pay compute cost once per unique config.

Hybrid Retrieval with RRF

For hybrid search, RAG Studio runs two systems simultaneously: ChromaDB for dense vector similarity and BM25Okapi (built in-memory from chunk text) for keyword sparse retrieval. The two ranked lists are fused with Reciprocal Rank Fusion: score = 1/(60+rank_dense) + 1/(60+rank_sparse). The constant k=60 makes fusion robust to outliers — no score normalization required.

Component Registry & Auto-UI

All loaders, splitters, and embedders self-register via Python decorators (@ComponentRegistry.register_loader) on import. Each component exposes a params_schema() class method that drives the dynamic UI forms — adding a new component requires only writing the class. The UI adapts automatically with no frontend changes needed.

Cross-Encoder Reranking

After initial retrieval, a cross-encoder re-reads the query and each candidate chunk together to produce a more precise relevance score. Unlike bi-encoders that encode query and passage independently, cross-encoders see both at once — capturing fine-grained interaction signals. Tradeoff: slower per query (k forward passes) but significantly higher precision when needed.

Auto-OCR Fallback

When text-based loaders return empty content (scanned PDF), the system falls back to PyMuPDF rendering: each page is rasterized to a high-DPI PNG (200–300 DPI, configurable), then Tesseract OCR extracts text character-by-character. No poppler dependency — PyMuPDF handles all PDF-to-image conversion, so Tesseract is the only external binary required.

Real-time Step Log

Pipeline execution runs in FastAPI BackgroundTasks. Each step writes its status and timing to a SQLite JSON step-log column. The frontend polls GET /api/pipelines/{id} every 2 seconds and renders live progress. An in-memory ring-buffer handler (last 500 records) also feeds the server Logs panel for diagnostics.
